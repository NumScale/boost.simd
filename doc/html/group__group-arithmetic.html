Arithmetic functions
User API » Functions and Constants

Description

Those functions provides scalar and SIMD algorithms for classical arithmetic operators and functions provided by the C and C++ standard library.

Other functions are also provided, in particular, provision for saturated operations through the use of a decorator.

  • Possibly saturated operations

    The functors:

    name name name name
    abs minus oneminusunary_minus
    dec multiplies sqr unary_plus
    dist plus toint touint
    inc

    can be decorated with the saturated_ decorator. This decorator has no effect on floating calls, but on integer calls replaces the operation by its saturated equivalent.

    Typically overflows will be replaced by the Valmin/Valmax proper value instead of providing undefined behaviour for signed integral types or wrapping modulo Valmax+1 for unsigned ones.

    Peculiarly saturated_(abs) and saturated_(dist) ensure that the result will never be stricly negative (which is for instance the case of abs(Valmin<T>()) for T being any signed integral type).

    toint is a rather common operation as it converts floating number to signed integers of the same bit size, nevertheless it probably is its saturated version you have to use because it acts properly on large or not finite values, this is why an alias for saturated_(toint) is provided as ifix.

  • Rounding operations

    name name name name
    ceil round ifloor inearbyint
    floor nearbyint ifix
    fix iceil iround

    -The operations prefixed by 'i' return a value of the integral type associated to the entry type. (If T is the entry type it is as_integer_t<T>)

    -The other ones return the same type as the entry.

  • Division operations

    divides is the function associated to standard division. There is another one which provides more flexibility, namely rounded divisions.

    With two parameters div and divides are equivalent, but div can admit a first option parameter that modifies its behaviour.

    The option parameter is described in the following table where a and b are of type T, fT is a supposed floating type associated to T (as_floating_t<T> if it exists) and iT is the integer type associated to T (as_integer_t<T>). (fT and iT are here only to support pseudo code description)

    option call result similar to
    ceil div(ceil, a, b) T(ceil(fT(a)/fT(b)))
    floor div(floor, a, b) T(floor(fT(a)/fT(b)))
    fix div(fix, a, b) T(fix(fT(a)/fT(b)))
    round div(round, a, b) T(round(fT(a)/fT(b)))
    nearbyint div(nearbyint, a, b) T(nearbyint(fT(a)/fT(b)))
    iceil div(iceil, a, b) iT(iceil(fT(a)/fT(b)))
    ifloor div(ifloor, a, b) iT(ifloor(fT(a)/fT(b)))
    ifix div(ifix, a, b) iT(ifix(fT(a)/fT(b)))
    iround div(iround, a, b) iT(iround(fT(a)/fT(b)))
    inearbyint div(inearbyint, a, b) iT(inearbyint(fT(a)/fT(b)))

  • Remainder operations

    rem is the remainder functor providing same kind of facilities as div

    With two parameters rem(a, b) is equivalent to rem(fix, a, b), but rem can admit a first optional parameter that modifies its behaviour and moreover can use the pedantic_ decorator to assure some limiting case values (see under).

    The option parameter can be chosen between ceil, floor, fix, round, nearbyint and if opt is the option, the call:

    rem(opt, a, b) is equivalent to a-b*div(opt, a, b)

    For floating entries the underlisted corner cases are handled in the following way:

    • if x is \(\pm\infty\) , Nan is returned
    • if x is \(\pm0\) and y is not 0 x is returned if pedantic_ is used (else 0: the sign bit is not preserved)
    • If y is \(\pm0\), Nan is returned
    • If either argument is a nan, a nan is returned

  • complex operations

    Boost.SIMD does not provides complex number operations yet, but it will soon. So the following functors that have a meaning as a restriction to real number of complex functions, can be seen as a prequel:

    name name name name name
    arg conj imag real sqr_abs

    For real entries conj and real are identity, imag always 0, sqr_abs coincide with sqr and arg results are always in the set \(\{0, \pi, Nan\}\)

  • Fused multiply-add operations

    name name name name
    fma fnma two_add two_split
    fms fnms two_prod

    Correct fused multiply/add implies

    • only one rounding
    • no "intermediate" overflow

    fnm? and fm? family provides this each time it is reasonable in terms of performance (mainly if the system has the hard wired capability).

    If you need "real" fma capabilities in all circumstances in your own code you can use pedantic_(fma) (although it can be expansive) or (generally still more expansive) std_(fma) by using the decorators.

    two_add, two_prod and two_split are used internally in pedantic_(fma) and can be useful in searching extra-accuracy in other circumstances as double-double computations.

    pedantic_(fma) is never used internally by Boost.SIMD

    • Standard operations

      The stdlibc++ provides them but only in scalar mode:

      name name name
      abs hypot remquo
      ceil max round
      floor min signbit
      fma rem (%) sqrt

      Boost.SIMD provides its own scalar and simd versions, but allows the use of the std_ decorator to call the associated system library function if the user needs it.

      average sqrt tenpower
      meanof sqr tofloat
      minmod sqrt1pm1
